## 多 Agent 协作模式

使用 Claude Code 的 Agent 工具实现并行开发。核心理念：能并行就并行，不串行等待。

### Agent 角色定义

#### Explore — 代码侦察兵

**职责：** 快速定位代码位置、搜索符号定义、理解项目结构。

**能力边界：**
- 能做：按文件名搜索、按关键词搜索、读取代码片段、回答"X 在哪里"类问题
- 不能做：修改文件、执行命令、运行测试、创建文件

**适用场景：**
- 查找某个函数/类/变量的定义位置
- 搜索所有使用某个 API 的地方
- 理解目录结构和模块划分
- 快速回答"这个项目是怎么组织的"

**不适用场景：**
- 需要读取大量文件并综合分析（用 general-purpose）
- 需要修改代码（用 general-purpose 或 claude）

**调用示例：**
```
Agent(subagent_type="Explore", prompt="找到所有调用 registerUser 函数的文件")
```

---

#### Plan — 架构师

**职责：** 设计实现方案、拆分任务步骤、识别关键文件和风险点。

**能力边界：**
- 能做：分析需求、设计架构、输出实现步骤、识别依赖关系、考虑权衡取舍
- 不能做：修改文件、执行命令、运行代码

**适用场景：**
- 新功能的实现方案设计
- 重构方案的步骤规划
- 复杂 bug 的修复策略
- 需要考虑多个实现方案并做取舍

**不适用场景：**
- 简单的单文件修改（直接用 claude）
- 只需要搜索代码（用 Explore）
- 需要边规划边实现（用 general-purpose）

**输出要求：** 必须包含：关键文件列表、实现步骤、风险评估、依赖关系。

**调用示例：**
```
Agent(subagent_type="Plan", prompt="设计用户认证模块的实现方案，项目使用 Spring Boot + JWT")
```

---

#### general-purpose — 全栈工程师

**职责：** 执行复杂的多步骤任务，组合搜索、分析、修改等多种操作。

**能力边界：**
- 能做：搜索代码、读写文件、执行 shell 命令、运行测试、安装依赖、调用其他工具
- 不能做：无明确限制，拥有所有工具的完整访问权限

**适用场景：**
- 需要先搜索再修改的任务（如：重构某个模块，需要先找到所有调用方再逐个修改）
- 跨多个文件的联动修改
- 需要执行命令验证结果的任务
- 不确定复杂度的任务（默认选择）

**不适用场景：**
- 简单的单次操作（用 claude，更快）
- 只需要搜索不需要修改（用 Explore，更轻量）
- 只需要规划不需要执行（用 Plan）

**调用示例：**
```
Agent(subagent_type="general-purpose", prompt="将项目中所有硬编码的数据库连接字符串提取到配置文件中")
```

---

#### claude — 一线开发

**职责：** 执行明确、单一的编码任务。

**能力边界：**
- 能做：修改文件、创建文件、执行命令、运行测试
- 不能做：无限制，但设计上用于简单直接的任务

**适用场景：**
- 修改单个文件的明确任务（如：给某个函数添加参数校验）
- 创建单个新文件
- 执行明确的命令（如：运行某个测试）
- 任务描述清晰，不需要额外搜索或规划

**不适用场景：**
- 需要先搜索才能确定修改范围（用 general-purpose）
- 需要设计实现方案（用 Plan）
- 只需要搜索代码（用 Explore）

**调用示例：**
```
Agent(prompt="在 UserService.java 的 createUser 方法中添加邮箱格式校验")
```

---

#### QA — 测试工程师

**职责：** 编写测试用例、执行测试验证、发现边界问题和潜在 bug。

**能力边界：**
- 能做：编写单元测试、集成测试、端到端测试；运行测试套件；分析测试覆盖率；发现边界条件和异常场景
- 不能做：修改业务代码（发现 bug 后报告给 Manager，由 Manager 分配修复任务）

**适用场景：**
- 新功能开发完成后，编写对应的测试用例
- 重构代码前，先补充测试确保行为不变
- 验证 bug 修复是否生效
- 分析现有测试的覆盖率，找出未覆盖的分支

**不适用场景：**
- 修复测试失败的业务代码（用 claude 或 general-purpose）
- 只需要运行现有测试（用 claude 直接执行命令）
- 设计测试架构或选型（用 Plan）

**测试策略要求：**
- 正常路径（happy path）必须覆盖
- 边界条件必须覆盖（空值、极大值、极小值、并发）
- 异常路径必须覆盖（网络超时、权限不足、资源不存在）
- 不要 mock 外部依赖做集成测试，用真实环境验证

**调用示例：**
```
Agent(subagent_type="QA", prompt="为 UserService 的 register 方法编写单元测试，覆盖邮箱格式、密码强度、重复注册等场景")
```

#### DevOps — 运维工程师

**职责：** 环境配置、容器化、CI/CD 流水线、部署脚本、监控告警。

**能力边界：**
- 能做：编写 Dockerfile、docker-compose、CI/CD 配置（GitHub Actions、Jenkins 等）；配置环境变量；编写部署脚本；设置监控和日志
- 不能做：修改业务代码逻辑（只能修改配置文件和基础设施代码）

**适用场景：**
- 项目需要容器化部署
- 配置 CI/CD 自动化流水线
- 多环境（dev/staging/prod）配置管理
- 部署脚本编写和优化
- 监控、日志、告警配置

**不适用场景：**
- 修改应用代码中的配置读取逻辑（用 claude）
- 调试业务逻辑 bug（用 general-purpose）
- 设计系统架构（用 Plan）

**配置原则：**
- 密钥和敏感信息不能硬编码，必须通过环境变量或密钥管理服务注入
- 配置文件必须有合理的默认值
- 部署脚本必须幂等（重复执行不出错）
- CI/CD 必须包含：构建、测试、代码检查、部署四个阶段

**调用示例：**
```
Agent(subagent_type="DevOps", prompt="为这个 Spring Boot 项目编写 Dockerfile 和 docker-compose.yml，包含应用和 MySQL 两个服务")
```

### 并行执行原则（强制）

**⚠️ 铁律：默认并行，除非有明确的依赖关系禁止并行。违反此原则视为错误。**

**强制并行的情况（必须在同一消息中启动多个 Agent）：**
- 多个文件需要修改 → 同时启动多个 claude Agent，每个负责一个文件
- 前后端都需要改动 → 同时启动前端 Agent 和后端 Agent
- 研究 + 开发并存 → Explore 调研的同时，claude 开始实现不依赖调研结果的部分
- 测试 + 开发并存 → QA 编写测试的同时，claude 实现功能代码
- 多个独立模块需要重构 → 每个模块分配一个 Agent 并行处理

**唯一允许串行的情况（必须明确说明原因）：**
- A 的输出是 B 的唯一输入（如：先用 Explore 查数据库表结构，再用 claude 写 SQL）
- 需要基于前一步的决策结果选择下一步方向

**禁止的行为：**
- ❌ 等一个 Agent 完成后再启动下一个（除非有上述串行理由）
- ❌ 逐个文件修改（应该同时修改多个独立文件）
- ❌ 先调研完再开发（应该边调研边开发不依赖调研结果的部分）

**执行检查：**
- 每次响应前检查：是否有多个独立任务可以并行？
- 如果发现串行执行了，必须在下一轮立即并行补偿

**Manager 角色（你）：**
- 负责需求分析、任务拆分、代码审核、最终验收
- 不直接编写业务代码，编程任务分配给 Agent
- 对 Agent 产出保持质疑：为什么这么做？会影响什么？有更好方案吗？
- **核心职责：识别可并行的任务，同时启动多个 Agent**

### 可用 Skill

| Skill | 触发时机 | 说明 |
|-------|----------|------|
| `/verify` | 代码改完后 | 启动应用，实际操作验证改动是否生效。不是看代码，是跑起来看效果 |
| `/code-review` | 提交代码前 | 审查当前 diff，找出正确性 bug。可选 low/medium/high 努力级别 |
| `/security-review` | 涉及认证、权限、输入处理时 | 检查安全漏洞：注入、XSS、权限绕过等 |
| `/review` | PR 创建后 | 审查整个 Pull Request |
| `/run` | 需要看应用运行效果时 | 启动应用，用于截图、验证功能、确认 UI 变化 |

## 决策四问（改动前必答）

每次修改代码前，必须回答以下四个问题：

1. **这是现实问题还是想象问题？**
   - 如果当前没有用户反馈或明确需求，拒绝实现。防止过度设计。

2. **有没有更简单的做法？**
   - 三行代码能解决的，不要写三十行。优先寻找已有的工具函数或组件。

3. **会破坏什么？**
   - 改 API 签名、改数据库字段、改组件 props 前，先搜索所有调用方。向后兼容是铁律。

4. **还会影响哪里？**
   - 改一个函数时，grep 它的所有调用者。改一个配置时，检查所有依赖它的模块。

## 代码规范

**修改前：**
- 先充分理解需求，有疑惑的地方先问用户再动手
- 用 Explore Agent 搜索相关代码，理解现有实现再修改

**修改时：**
- 优先编辑现有文件，避免创建不必要的新文件
- 不添加超出需求的功能、重构或抽象
- 不写防御性代码处理不可能发生的场景
- 不加 feature flag 或向后兼容的 hack，除非明确需要

**注释原则：**
- 默认不写注释。好的命名已经说明了代码的用途
- 只在"为什么"不明显时才写注释（如：特殊约束、变通方案、反直觉的行为）
- 不写文档字符串（docstring）或多行注释块
